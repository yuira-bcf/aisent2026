## 1. 設計方針

### 1.1 再設計の背景

04_アーキテクチャ設計書で提案された4サービス分離構成（Core API / AI Service / EC Service / Notify Service）は運用コストが高く、Phase 1の規模に対して過剰である。本設計書では**2サービス構成**に簡素化し、最小限のインフラで全機能をカバーする。

### 1.2 選択方針: Option C（2言語ハイブリッド）

| 方針 | 内容 |
|------|------|
| コアサービス | **Next.js 15** で認証・CRUD・EC・通知を一括処理 |
| AIサービス | **Python FastAPI** でAI処理を分離（Phase 1.5導入） |
| 簡素化 | 4サービス → 2サービスで運用コストを大幅削減 |
| 適材適所 | AI領域はPythonエコシステムを活用 |

### 1.3 主要設計判断

| 判断 | 選択 | 理由 |
|------|------|------|
| REST API | **廃止** | Server Componentsが直接DB参照、Server Actionsが変更処理。REST層不要 |
| 状態管理 | **なし** | Server Components + useActionState + URL state。Redux/Zustand不要 |
| DB分離 | **単一DB、スキーマ分離なし** | Next.jsが全テーブルにアクセスするため分離は無意味 |
| Phase 1のAI | **Vercel AI SDK（暫定）** | Python AI Serviceなしで最速リリース。Phase 1.5でPythonに差替え |

## 5. 認証・認可設計

### 5.1 Auth.js v5 + JWT Strategy

```
ユーザー → Credentials Provider → JWT発行（role含む）
                                    → middleware.tsでルート制御
                                    → Server ActionsでロールチェックHttpOnly Cookie
```

- Phase 1: Credentials Provider（メール/パスワード）
- Phase 3: Google / Apple Provider追加
- JWT Claims: `{ id, email, name, role }`
- ロール: `CUSTOMER` / `CREATOR` / `ADMIN`

### 5.2 3層アクセス制御

| 層 | 場所 | 役割 |
|----|------|------|
| 1 | `middleware.ts` | 未認証→/login、ロール不足→リダイレクト |
| 2 | Server Actions | 各mutation冒頭でsession + role検証 |
| 3 | Server Components | ロール別UI出し分け（表示制御のみ） |

### 5.3 AI Service認証

Next.js → Python は共有シークレット（`INTERNAL_SERVICE_TOKEN`）のBearer認証。ユーザートークンは不要。

## 10. 検証方法

### 10.1 Phase 1完了時の検証項目

| 項目 | 内容 |
|------|------|
| 調合計算の正確性 | 同一入力（キーワード+重み+ルール+比率）で、Java実装と同一の出力比率を得ることをVitestで検証 |
| E2Eフロー | Playwrightで「登録→ログイン→キーワード選択→比率設定→調合生成→結果表示→命名→履歴確認」を自動テスト |
| クリエーターフロー | 「ログイン(CREATOR)→キーワード一覧→ルール編集→AI提案→保存→統計確認」 |
| 認可テスト | CUSTOMER/CREATOR/ADMINの各ロールで適切なアクセス制御が機能することを確認 |
| AIフォールバック | OPENAI_API_KEY未設定時にテンプレートベースのストーリー・ルール提案が動作 |
| TypeScriptビルド | `tsc --noEmit` エラーゼロ |
| Docker | `docker compose up` で全サービス起動・動作確認 |

### 11.3 ai-client.ts のモデル選択ロジック

```typescript
// lib/ai-client.ts
import { env } from '@/lib/env';
import { openai } from '@ai-sdk/openai';
import { anthropic } from '@ai-sdk/anthropic';

function getModel(modelId: string) {
  if (modelId.startsWith('gpt-')) {
    return openai(modelId);
  }
  if (modelId.startsWith('claude-')) {
    return anthropic(modelId);
  }
  throw new Error(`Unsupported model: ${modelId}`);
}

export const primaryModel = getModel(env.AI_MODEL);
export const fallbackModel = getModel(env.AI_FALLBACK_MODEL);
```
