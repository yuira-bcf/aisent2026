## 1. アーキテクチャ方針

### 1.1 設計原則

| 原則 | 内容 |
|------|------|
| 最小サービス構成 | 2サービス（Next.js + Python AI Service）で全機能をカバー |
| REST API廃止 | Server Components/Actions で REST 層を排除し、開発・保守コストを削減 |
| 適材適所 | AI領域のみPython、それ以外はTypeScriptで統一 |
| 段階的拡張 | Phase 1はNext.jsのみ、Phase 1.5でPython AI Service追加 |
| フルスタック型安全 | TypeScript strict + Zod + Drizzle推論型で型安全を保証 |

### 1.2 旧設計からの変更点

| 旧設計（v2.0） | 新設計（v3.0） | 変更理由 |
|---------------|---------------|---------|
| 4サービス分離（Core API/AI/EC/Notify） | 2サービス（Next.js + Python AI） | Phase 1の規模に対して4サービスは過剰。運用コスト削減 |
| API Gateway (nginx/OpenResty) | 不要 | Next.jsが全リクエストを直接処理。サービス間通信は内部のみ |
| Spring Boot (Core API + EC Service) | Next.js (Server Components/Actions) | REST API層の排除。フルスタックTypeScriptで開発効率向上 |
| Node.js Notify Service | Next.js内のResend + React Email | 独立サービスにするほどの複雑性がない |
| DB スキーマ分離（core/ec） | 単一DB、分離なし | Next.jsが全テーブルにアクセスするため分離は無意味 |
| Celery + Redis キュー | FastAPI BackgroundTasks | Phase 1は2エンドポイント。Celeryは過剰 |
| LangChain | Jinja2テンプレート | Phase 1は4テンプレートのみ。LangChainは過剰 |

## 3. Next.js アーキテクチャ詳細

### 3.1 Server Components / Client Components / Server Actions

| レイヤー | 役割 | 用途 |
|---------|------|------|
| Server Components | データ読取 | DB直接クエリ、ページレンダリング |
| Client Components | インタラクション | フォーム、スライダー、チャート |
| Server Actions | データ変更 | CRUD操作、認証、AI呼出し |
| Route Handlers | 外部通信 | Webhook受信、AIコールバック |

### 3.2 レンダリング戦略

| ページ | レンダリング | 理由 |
|--------|------------|------|
| /login, /register | Client Component | フォームインタラクション |
| /blend | Server + Client島 | プリセット取得はServer、フォームはClient |
| /result/[id] | SSR | OGP/SNS共有対応 |
| /history | Server Component | DB直接クエリ |
| /creators (Phase 2) | ISR 60秒 | SEO + キャッシュ |
| /creators/[id] (Phase 2) | ISR 60秒 | SEO |
| /keywords, /stats | Server Component | 管理画面、SSR不要 |

### 3.3 認証・認可（Auth.js v5）

#### 3層アクセス制御

| 層 | 場所 | 役割 |
|----|------|------|
| 1 | `middleware.ts` | 未認証→/login、ロール不足→リダイレクト |
| 2 | Server Actions | 各mutation冒頭でsession + role検証 |
| 3 | Server Components | ロール別UI出し分け（表示制御のみ） |

#### ルーティング制御

```typescript
// middleware.ts
const publicPaths = ['/login', '/register'];
const creatorPaths = ['/keywords', '/stats'];
const adminPaths = ['/admin'];

// 未認証 → /login
// CUSTOMER が /keywords → リダイレクト
// CUSTOMER/CREATOR が /admin → リダイレクト
```

### 3.4 データアクセスパターン

```
Server Component (読取)
  → Drizzle ORM → PostgreSQL
  → セッション情報で自動フィルタ

Server Action (変更)
  → セッション検証
  → Zodバリデーション
  → Drizzle ORM → PostgreSQL
  → revalidatePath() でキャッシュ無効化
  → redirect() で遷移

Route Handler (外部)
  → ヘッダー認証（INTERNAL_SERVICE_TOKEN）
  → DB更新
  → 200 OK
```

| 機能 | OpenAI失敗時の代替動作 |
|------|---------------------|
| ストーリー生成 | テンプレートベースの日本語テキスト生成 |

### 4.4 Phase 1暫定: Vercel AI SDK

Phase 1ではPython AI Serviceを構築せず、Next.js内で `ai` + `@ai-sdk/openai` パッケージを使いOpenAIを直接呼出す。`ai-client.ts` のインターフェースを統一しておき、Phase 1.5でPythonに差替え時にこのファイルのみ変更。

## 5. サービス間通信設計

### 5.1 通信パターン

| パターン | 用途 | 実装 |
|---------|------|------|
| Next.js → AI Service | ストーリー生成、ルール提案 | HTTP (Bearer INTERNAL_SERVICE_TOKEN) |
| AI Service → Next.js | 非同期結果コールバック | HTTP POST /api/internal/... |
| 外部 → Next.js | Stripe Webhook (Phase 3) | HTTPS POST /api/webhooks/stripe |

### 5.2 AI Service認証

```
Next.js → Python AI Service:
  Authorization: Bearer {INTERNAL_SERVICE_TOKEN}
  → 共有シークレットによるサービス間認証
  → Docker内部ネットワークで直接通信
  → ユーザートークンは不要
```

### 5.3 非同期処理フロー

```
1. ユーザー: 調合リクエスト送信
2. Next.js Server Action:
   a. 調合計算（blend-calculator.ts、同期）
   b. 結果保存（story=null）
   c. AI Service呼出し（非同期 or Phase 1は同期）
3. AI Service:
   a. OpenAI API呼出し
   b. 成功: Next.js Callback → story更新
   c. 失敗: フォールバック結果を返却
4. Next.js: redirect → 結果ページ表示
   (storyがnullの場合はローディング表示 → ポーリングで更新)
```

## 8. 段階的実装計画

### Phase 1 → Phase 1.5 移行

```
Phase 1（Next.js + PostgreSQL）
  → Vercel AI SDK で OpenAI 直接呼出し

Phase 1.5（+ Python AI Service）
  Step 1: FastAPI プロジェクト作成
  Step 2: ストーリー生成エンドポイント + Jinja2プロンプト
  Step 3: ルール提案エンドポイント
  Step 4: フォールバックロジック移植
  Step 5: ai-client.ts をPython呼出しに差替え
  Step 6: Docker Compose に ai-service + redis 追加

  影響範囲: ai-client.ts のみ（インターフェース不変）
  リスク: 低
```

### Phase 2（クリエーター・マーケットプレイス）

```
  Step 1: creator_profiles, creator_favorites テーブル追加
  Step 2: blend_requests.creator_id カラム追加
  Step 3: クリエーター画面実装（ISR、SEO対応）
  Step 4: 調合計算の creator_id フィルタリング
  Step 5: 管理者画面（承認フロー）

  影響範囲: Next.js + Drizzleスキーマ
  リスク: 中（既存ロジック変更あり）
```

### Phase 3（コマース）

```
  Step 1: EC系テーブル追加（products, orders, payments等）
  Step 2: Stripe連携（Route Handler でwebhook受信）
  Step 3: 注文管理画面
  Step 4: Resend + React Email でメール通知

  影響範囲: Next.js + Drizzleスキーマ
  リスク: 中（決済処理のセキュリティ考慮）
```

### Phase 4（パーソナライゼーション）

```
  Step 1: Python AI Service に嗜好分析エンドポイント追加
  Step 2: クリエーター推薦エンドポイント追加

  Step 4: レコメンドUI実装

  影響範囲: Python AI Service + Next.js
  リスク: 低（新規追加のみ）

## 9. メリット・トレードオフ

### 9.1 メリット

| 観点 | メリット |
|------|---------|
| **開発効率** | TypeScript 1言語でフルスタック開発。REST API層不要 |
| **運用コスト** | 2サービスのみ。監視・ログ管理が容易 |
| **型安全** | Drizzle推論型 + Zod + TypeScript strictで統一的な型安全 |
| **AI最適化** | PythonでAI/MLエコシステムを最大活用 |
| **パフォーマンス** | Server Components でゼロバンドルサイズ、SSR/ISR対応 |
| **SEO** | Next.js のSSR/ISR/SSGでクリエーターページ等のSEO最適化 |
| **段階的拡張** | Phase 1はNext.jsのみで最速リリース |

### 9.2 トレードオフ

| 観点 | トレードオフ | 軽減策 |
|------|------------|--------|
| **スケーリング粒度** | AI処理とWeb処理が同一サービスに混在 | Phase 1.5でAI分離済み。Phase 4でさらに分離可能 |
| **決済セキュリティ** | 決済処理が一般ロジックと同一プロセス | Route Handlerで分離、Stripe Checkoutによるクライアント側決済 |
| **2言語運用** | TypeScript + Python の2言語 | AIはPython必須、それ以外はTypeScript統一 |
| **Vercel依存リスク** | Next.js はVercelとの親和性が高い | Docker Compose でセルフホスト可能 |

```

### 12.2 バージョンルール

| 変更種別 | バージョン影響 | 例 |
|---------|-------------|-----|
| フィールド追加（レスポンス） | なし（v1内） | レスポンスに `createdAt` フィールド追加 |
| オプショナルパラメータ追加 | なし（v1内） | クエリパラメータ `?sort=newest` 追加 |
| フィールド削除 | **メジャーバージョンアップ** | レスポンスから `legacyField` 削除 → v2 |
| フィールドリネーム | **メジャーバージョンアップ** | `user_name` → `displayName` → v2 |
| 型変更 | **メジャーバージョンアップ** | `price: string` → `price: number` → v2 |
| エンドポイント削除 | **メジャーバージョンアップ** | `/api/v1/legacy` 廃止 → v2 移行 |

### 12.3 サポートポリシー

| ポリシー | 内容 |
|---------|------|
| 新バージョン公開時 | 旧バージョンは**6ヶ月間**サポート継続 |
| 非推奨通知 | レスポンスヘッダー `Deprecation: true` + `Sunset: <date>` を付与 |
| サポート終了時 | 旧バージョンは `410 Gone` を返却 |
