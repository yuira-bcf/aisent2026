## 1. プロジェクト概要

### 1.1 ビジョン
調香師の感性をAI化し、ユーザーに合った香りをオーダーメイドで調合・販売するプラットフォーム。複数の調香師（クリエーター）がそれぞれ独自のルールセットを持ち、同じキーワードでもクリエーターによって異なる調合結果を生み出す。ユーザーはオーダーを重ねることで自分の嗜好に合った香りへと洗練させていくことができる。

### 1.2 ビジネスモデル
| 項目 | 内容 |
|------|------|
| 事業形態 | オーダーメイド香水のC2Cプラットフォーム |
| 収益モデル | 受注手数料 + クリエーターへのロイヤリティ支払い |
| 注文単位 | 1本単位での注文が可能 |
| 受注管理 | KyaraInnovate社が全受注・配送を一括管理 |
| クリエーター報酬 | 売上に応じたロイヤリティをクリエーターに支払い |

### 1.3 背景
- 香水調合は専門知識が必要で、一般ユーザーにはハードルが高い
- 調香師の感性・ノウハウをAIルールとして体系化し、スケーラブルに活用する
- キーワードベースのインターフェースにより、感覚的な香り選択を可能にする
- OpenAI APIを活用し、ストーリー生成やルール提案を自動化する
- 個々の調香師の「個性」を商品差別化の源泉とする

### 1.4 コンセプト
```
調香師A「夏」→ 爽やかな柑橘系ブレンド
調香師B「夏」→ トロピカルなフローラルブレンド
→ 同じキーワードでも、クリエーターの感性で全く異なる香りが生まれる
```

### 1.5 対象ユーザー

| ロール | 説明 |
|--------|------|
| CUSTOMER（顧客） | 香りの調合リクエストを作成し、注文・購入する |
| CREATOR（クリエーター / 調香師） | 自身の感性をルールセットとして定義・管理する |
| ADMIN（管理者） | プラットフォーム全体の管理、クリエーター承認、売上管理を行う |

> **Note**: 現システムの `PERFUMER` ロールを `CREATOR` に拡張。クリエーターは調香師に限定されず、独自の感性でルールを作成できるプロデューサー的存在。

## 2. フェーズ計画

### フェーズ1: コア機能（実装済み）
キーワード/手動選択による調合生成、ルール管理、AI連携の基盤システム。

### フェーズ2: クリエーター・マーケットプレイス
複数クリエーター対応、クリエーター選択、プロフィール機能。

### フェーズ3: コマース（注文・販売）
商品化、注文、決済、ロイヤリティの仕組み。

### フェーズ4: パーソナライゼーション
ユーザー嗜好学習、レコメンド、リピート最適化。

## 4. 非機能要件

### 4.1 性能要件

| 項目 | フェーズ1 | フェーズ3以降 |
|------|---------|-------------|
| レスポンス時間 | API応答: 1秒以内（AI生成除く） | 同左 |
| AI生成時間 | ストーリー/ルール提案: 5〜15秒 | 同左 |
| 同時接続 | 10ユーザー（開発フェーズ） | 100ユーザー以上 |
| 決済処理 | - | 3秒以内 |

### 4.2 セキュリティ要件

| 項目 | 要件 |
|------|------|
| 認証 | JWT (HS256)、有効期限24時間 |
| パスワード | BCrypt ハッシュ化、最低6文字 |
| 認可 | ロールベースアクセス制御 |
| データ分離 | ユーザーは自身の調合結果・注文のみ閲覧可能 |
| CSRF | ステートレスJWTのため無効化 |
| CORS | 本番ドメインのみ許可 |
| 決済データ | PCI DSS準拠（外部決済サービス利用により対応）|
| 個人情報 | 配送先住所等は暗号化保存 |

### 4.3 可用性要件

| 項目 | 要件 |
|------|------|
| OpenAI障害時 | フォールバック機能で基本動作を継続 |
| DB | PostgreSQL 16、永続化ボリューム |
| 決済障害時 | リトライ機構、未決済注文の保留管理 |

### 4.4 技術制約

| 項目 | 要件 |
|------|------|
| バックエンド | Next.js 15 (App Router), TypeScript 5 |
| フロントエンド | React 19, Tailwind CSS v4 |
| AI | Python FastAPI + claude-sonnet-4.5（ハイブリッド構成） |
| データベース | PostgreSQL 16, Drizzle ORM |
| インフラ | Docker Compose（開発）→ Vercel + クラウド（本番） |
| 決済 | Stripe (Checkout Sessions) |
| メール | Resend + React Email |

## 7. ユーザーフロー

### 7.1 基本調合フロー（フェーズ1 - 実装済み）
```
ログイン → キーワード選択 → ノート比率設定 → (香料手動選択)
→ 調合生成 → 結果表示（円グラフ+ストーリー） → フレグランス命名
```

### 7.2 クリエーター選択フロー（フェーズ2）
```
ログイン → クリエーター一覧 → クリエーター詳細（スタイル確認）
→ 「このクリエーターで調合する」 → キーワード選択 → ノート比率設定
→ 調合生成（選択クリエーターのルールのみ使用）→ 結果表示
```

### 7.3 注文フロー（フェーズ3）
```
調合結果 → 「この香りを商品化する」 → 商品名・説明設定
→ サイズ選択（30ml/50ml/100ml）→ 配送先選択 → 決済
→ 注文確定 → 製造 → 発送 → 完了
```

### 7.4 リピートフロー（フェーズ4）
```
注文履歴 → 「この香りを再調合する」 → 前回の設定がプリセット
→ キーワード重み微調整 → 調合生成 → 比較表示 → 注文
```

## 9. ビジネスルール

### 9.1 調合ルール
| ルール | 内容 |
|--------|------|
| ノート比率合計 | TOP + MIDDLE + LAST = 100% |
| ノート内比率合計 | 各ノート内の香料比率合計 = 100% |
| ルール重み範囲 | 0.00 〜 1.00（0.05単位） |
| キーワード重み範囲 | 0 〜 10（整数） |

### 9.2 クリエータールール（フェーズ2）
| ルール | 内容 |
|--------|------|
| ルール所有権 | ルールはクリエーターごとに独立（同じキーワードでも別ルール） |
| 調合時のルール選択 | 選択されたクリエーターのルールのみを使用 |
| 最低カバレッジ | クリエーター公開にはプリセットキーワードの50%以上にルール定義が必要 |
| クリエーター承認 | 管理者による審査・承認が必要 |

### 9.3 注文ルール（フェーズ3）
| ルール | 内容 |
|--------|------|
| 最低注文数 | 1本 |
| 注文キャンセル | 製造開始前のみ可能 |
| ロイヤリティ率 | 売上の 10〜30%（管理者設定、クリエーターランクに応じて変動） |
| ロイヤリティ支払い | 月末締め翌月払い |
| 価格設定 | サイズ × 基本単価 + クリエーター補正（将来的に検討） |

## 10. 用語定義

| 用語 | 説明 |
|------|------|
| ノート | 香水の香りの段階。TOP（トップ）、MIDDLE（ミドル）、LAST（ラスト）の3段階 |
| TOPノート | 最初に感じる軽い香り（柑橘系・フレッシュ系）|
| MIDDLEノート | 中心となる香り（フローラル系）|
| LASTノート | 持続する深い香り（ウッディ系・ムスク系）|
| 香料（フレーバー） | 香水を構成する個別の香り成分 |
| ルール | キーワードと香料の対応関係（重み付き）。クリエーターごとに独立 |
| クリエーター | 自身の感性をルールセットとして定義し、AIを通じて調合を提供する人 |
| 重み（weight） | キーワードの重要度(0-10) またはルールの関連度(0-1.0) |
| 比率（ratio） | ノート間の配分比率、またはノート内の香料配分比率 |
| カバレッジ | ルールが定義済みのキーワードの割合 |
| ロイヤリティ | クリエーターに支払われる売上連動型報酬 |
| 商品化 | 調合結果に名前をつけ、注文可能な商品として確定すること |
| オーダーメイド | ユーザーのキーワード指定に基づく一点もの調合 |

## 12. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| OpenAI API障害 | 中 | フォールバック機能（実装済み）|
| クリエーター不足 | 高 | 初期はAI一括生成でルール作成のハードルを下げる（実装済み）|
| 決済トラブル | 高 | Stripe等の実績あるサービスを利用、テスト環境での十分な検証 |
| 個人情報漏洩 | 高 | 暗号化保存、最小権限の原則、定期的なセキュリティ監査 |
| クリエーター品質 | 中 | 承認制 + 最低カバレッジ要件で品質担保 |
| 単品注文の製造コスト | 高 | 最低価格設定、バッチ製造の最適化 |
