# KyaraInnovate Architecture Review Framework (KARF)

> システム品質を体系的に検証するためのアーキテクチャレビューフレームワーク

最終更新: 2026-02-14

---

## 1. 概要

### 1.1 目的

KyaraInnovate の設計・実装の整合性と品質を、業界標準の手法を組み合わせて網羅的に検証する。

### 1.2 採用手法

| 手法 | 役割 | 適用範囲 |
|---|---|---|
| **C4 モデル** | 構造検証の骨格 | 4つの抽象レベルで設計と実装の乖離を検出 |
| **ISO 25010** | 品質特性の網羅性担保 | 8品質特性 × C4レベルのマトリクス評価 |
| **FMEA** | リスク定量化 | Warn/Fail 箇所の故障モード分析・優先度付け |
| **ATAM** | トレードオフ分析 | アーキテクチャ判断点のシナリオベース検証 |

### 1.3 フレームワーク全体構造

```
┌─────────────────────────────────────────────────────────┐
│  Phase 1: C4 構造検証                                    │
│  4つの抽象レベルで設計と実装の乖離を検出                    │
├─────────────────────────────────────────────────────────┤
│  Phase 2: ISO 25010 品質特性レビュー                      │
│  8品質特性 × C4レベル のマトリクスで網羅的に評価             │
├─────────────────────────────────────────────────────────┤
│  Phase 3: FMEA リスク分析                                │
│  Phase 2 で Warn/Fail の箇所に故障モード分析を適用          │
├─────────────────────────────────────────────────────────┤
│  Phase 4: ATAM シナリオ分析                               │
│  アーキテクチャ上のトレードオフ判断点を検証                   │
└─────────────────────────────────────────────────────────┘
```

### 1.4 手法選定の根拠

| 手法 | 強み | 弱み | KyaraInnovate 適合度 |
|---|---|---|---|
| ATAM | 品質属性トレードオフの可視化、ステークホルダー合意 | 重厚（本来は複数日ワークショップ）、個人開発には過剰 | △ エッセンスのみ採用 |
| C4 + チェックリスト | 抽象度4段階で構造を可視化、検証ポイントが明確 | 品質特性の網羅性が弱い | ◎ 構造検証に最適 |
| ISO 25010 | 8品質特性で網羅的、国際標準で客観的 | 抽象的で具体チェック項目への落とし込みが必要 | ○ 品質軸の骨格に採用 |
| FMEA | リスク定量化（RPN）、優先度が明確 | 全故障モード列挙が膨大、確率推定が主観的 | ○ 重要パスに限定適用 |

## 2. Phase 1: C4 構造検証

4つの抽象レベルで「設計書 vs 実装」の乖離を検出する。

### 2.1 L1 Context（システム境界）

| # | チェック項目 | 検証対象 | 検証方法 |
|---|---|---|---|
| C1.1 | 外部システム一覧が設計書に定義済み | Stripe / Resend / Sentry / Redis / PostgreSQL | アーキテクチャ設計書の外部システム図を確認 |
| C1.2 | 外部システムとの通信プロトコルが明記 | HTTPS / WebSocket / SMTP | 設計書に通信方式の記載があるか |
| C1.3 | ユーザー種別（ペルソナ）が定義済み | Consumer / Creator / Admin | 要件定義書のアクター一覧と roleEnum の一致 |
| C1.4 | システム境界のデータフローが文書化 | 入出力データの型と方向 | 設計書のデータフロー図の存在と正確性 |
| C1.5 | 外部システム障害時のフォールバックが定義済み | 各外部依存の縮退動作 | operations-infrastructure.md の該当記述 |

### 2.2 L2 Container（コンテナ構成）

| # | チェック項目 | 検証対象 | 検証方法 |
|---|---|---|---|
| C2.1 | コンテナ一覧と責務が定義済み | Next.js App / PostgreSQL / Redis / AI Service | docker-compose.yml vs アーキテクチャ設計書 |
| C2.2 | コンテナ間通信が設計書と一致 | ポート / プロトコル / 認証 | docker-compose.yml のネットワーク設定確認 |
| C2.3 | 各コンテナのスケーリング戦略が定義済み | Vercel serverless / DB connection pool | operations-supplement.md §2 確認 |
| C2.4 | コンテナ間の認証方式が定義済み | Internal token / API key | env.ts の INTERNAL_SERVICE_TOKEN 等 |
| C2.5 | 環境別構成（dev/staging/prod）が定義済み | 環境変数 / 接続先 | .env.example と env.ts の一致 |

### 2.3 L3 Component（コンポーネント構成）

| # | チェック項目 | 検証対象 | 検証方法 |
|---|---|---|---|
| C3.1 | サービス層の責務分離が適切 | services/*.ts の凝集度 | 1サービス = 1ドメインの原則確認 |
| C3.2 | スキーマ → サービス → API ルートの対応が完全 | 3層の突合 | schema テーブル数 vs サービス関数数 vs API ルート数 |
| C3.3 | 共通ユーティリティの重複がない | lib/api/*.ts の整理状態 | コード重複検出 |
| C3.4 | 画面 → API → サービスのデータフローが追跡可能 | mockup → route → service | go() 遷移先と API 呼出の対応 |
| C3.5 | 循環依存がない | import グラフ | import 文の依存方向分析 |
| C3.6 | ミドルウェアが適切に適用 | auth-guard / rate-limiter / security-headers | 適用対象エンドポイントの網羅性 |

### 2.4 L4 Code（コードレベル）

| # | チェック項目 | 検証対象 | 検証方法 |
|---|---|---|---|
| C4.1 | 型定義と DB スキーマが一致 | TypeScript types vs Drizzle schema | 型フィールド vs カラム定義の突合 |
| C4.2 | Zod スキーマと DB 制約が一致 | バリデーション vs CHECK/NOT NULL | min/max/regex の突合 |
| C4.3 | エクスポート漏れがない | index.ts vs 各モジュール定義 | export 一覧 vs 定義一覧の差分 |
| C4.4 | 命名規則の一貫性 | camelCase / snake_case | DB: snake_case, TS: camelCase の徹底 |
| C4.5 | デッドコードがない | 未使用 export / import | 参照のない定義の検出 |
| C4.6 | エラーハンドリングが統一的 | throw vs return vs apiError | 既存パターン (safeHandler + apiError) 準拠 |

## 3. Phase 2: ISO 25010 品質特性レビュー

8品質特性を C4 レベル別に評価する。各セルに **Pass(P) / Warn(W) / Fail(F) / N/A** を記入。

### 3.1 評価マトリクス

| ISO 25010 品質特性 | Context | Container | Component | Code |
|---|---|---|---|---|
| **機能適合性** | 要件カバレッジ | 機能配置の適切性 | CRUD 網羅性 | Zod バリデーション |
| **性能効率性** | SLO 定義 | キャッシュ戦略 | N+1 クエリ | Index 定義 |
| **互換性** | API バージョニング | コンテナ間 I/F | モジュール間契約 | 型互換性 |
| **使用性** | UX フロー完全性 | レスポンス形式統一 | エラーメッセージ | 空状態/ローディング |
| **信頼性** | DR/RPO/RTO | ヘルスチェック | リトライ/フォールバック | ゼロ除算ガード |
| **セキュリティ** | 脅威モデル | 認証/認可 | RBAC 適用 | 入力サニタイズ |
| **保守性** | ドキュメント同期 | CI/CD | テストカバレッジ | コード規約 |
| **移植性** | 環境変数管理 | コンテナ化 | DI/抽象化 | ハードコード排除 |

### 3.2 品質特性別の詳細チェック

#### 3.2.1 機能適合性 (Functional Suitability)

| 副特性 | チェック内容 | 判定基準 |
|---|---|---|
| 機能完全性 | 設計書の全機能に対応実装がある | F: 実装率<80%, W: <95%, P: ≥95% |
| 機能正確性 | 計算ロジックが仕様通り | F: 不一致あり, P: 一致 |
| 機能適切性 | 不要な機能がない (YAGNI) | W: 違反あり, P: 適切 |

検証ポイント:
- 設計書の機能一覧 vs サービス関数の突合
- 調合計算 (blend-calculator.ts) のロジックが設計書 (blend-system.md) と一致
- ロイヤリティ計算がモード別に正しい (ai_only=0% / styled=70% / recipe=100%)
- 全 CRUD 操作の網羅性チェック（作成あり削除なし等のパターン検出）

#### 3.2.2 性能効率性 (Performance Efficiency)

| 副特性 | チェック内容 | 判定基準 |
|---|---|---|
| 時間効率性 | SLO が定義済み + 達成可能 | F: 未定義, W: 定義のみ, P: 検証済み |
| 資源効率性 | 接続プール / キャッシュ設計 | F: 未設計, W: 設計のみ, P: 実装済み |
| 容量満足性 | 負荷テスト計画がある | F: 未計画, W: 計画のみ, P: 実施済み |

検証ポイント:
- SLO 定義: p95 <500ms (read), <2s (blend), <3s (checkout)
- Redis キャッシュ TTL 設定の妥当性
- DB Index がクエリパターンに適合
- N+1 クエリの有無（relations を使用した JOIN の適切性）
- DB 接続プールの設定値妥当性 (min 2, max 10)

#### 3.2.3 互換性 (Compatibility)

| 副特性 | チェック内容 | 判定基準 |
|---|---|---|
| 共存性 | 外部サービス障害時のフォールバック | F: 未考慮, P: 設計済み |
| 相互運用性 | API レスポンス形式が統一 | F: 不統一, P: ApiResponse 統一 |

検証ポイント:
- API バージョニング (/api/v1/) の一貫性
- ApiResponse 型 (ok + data / error + message) の統一使用
- Stripe / Resend / Sentry 障害時の縮退動作定義

#### 3.2.4 使用性 (Usability)

| 副特性 | チェック内容 | 判定基準 |
|---|---|---|
| 操作性 | 全画面フローが遷移可能 | F: 孤立画面あり, P: 全遷移可 |
| エラー防止 | バリデーション + 確認ダイアログ | W: 一部欠如, P: 網羅 |
| UI 一貫性 | デザインシステム準拠 | W: 逸脱あり, P: 準拠 |

検証ポイント:
- mockup-mobile.html の全 go() 呼出先に対応する screen が存在
- Drawer メニューの全項目に対応画面がある
- 画面間の戻るボタンが正しい遷移元を指定
- 空状態 (データ0件時) の表示が定義済み
- ロール別 (Consumer/Creator/Admin) のアクセス制御が画面に反映

#### 3.2.5 信頼性 (Reliability)

| 副特性 | チェック内容 | 判定基準 |
|---|---|---|
| 成熟性 | 既知のバグ / 設計欠陥がない | F: 致命的欠陥, W: 軽微, P: なし |
| 可用性 | ヘルスチェック + 監視 | F: 未実装, W: 設計のみ, P: 実装済み |
| 障害許容性 | 部分障害時の縮退運転 | F: 未考慮, W: 設計のみ, P: 実装済み |
| 回復性 | バックアップ + DR 手順 | F: 未定義, W: 定義のみ, P: 検証済み |

検証ポイント:
- RPO: 1時間, RTO: 4時間の定義と達成手段
- /api/health, /api/health/ready の実装と外部監視連携
- Redis 障害時のフォールバック（レート制限のフェイルオープン）
- Stripe Webhook の冪等性処理
- バッチジョブの排他制御（Redis SETNX）と障害時リトライ

#### 3.2.6 セキュリティ (Security)

| 副特性 | チェック内容 | 判定基準 |
|---|---|---|
| 機密性 | 認証 + RBAC + データ暗号化 | F: 認証欠如, W: 部分的, P: 完備 |
| 完全性 | 入力バリデーション + CSRF 対策 | F: 未対策, W: 部分的, P: 完備 |
| 否認防止 | 監査ログ | F: 未実装, W: 設計のみ, P: 実装済み |
| 責任追跡性 | ユーザー操作の追跡可能性 | F: 追跡不可, P: 追跡可能 |

検証ポイント:
- requireAuth() が全保護エンドポイントに適用
- レート制限: 100 req/min (general), 10/min (auth), 5/min (password reset), 30/min (blend)
- 監査ログ: 13 アクションタイプの網羅性
- CSP / X-Frame-Options / HSTS の設定
- パスワードリセットトークン: SHA-256 + 1時間有効期限 + 定数時間比較
- DOMPurify による UGC サニタイズ

#### 3.2.7 保守性 (Maintainability)

| 副特性 | チェック内容 | 判定基準 |
|---|---|---|
| モジュール性 | 関心の分離 (schema/service/api) | W: 責務混在, P: 分離済み |
| 再利用性 | 共通ユーティリティの活用 | W: 重複コード, P: 共通化済み |
| 解析性 | 構造化ログ + エラートラッキング | F: 未実装, W: 設計のみ, P: 実装済み |
| 変更性 | テスト + CI/CD | F: テストなし, W: 部分的, P: 完備 |
| テスト容易性 | テスト可能な設計 | F: テスト不可, W: 部分的, P: 完備 |

検証ポイント:
- 3層分離: packages/db (schema) → apps/web/lib/services → apps/web/app/api
- CI/CD パイプライン (.github/workflows/ci.yml) の完全性
- pino による構造化ログの設計
- Drizzle Kit マイグレーション戦略
- 設計書の変更ログが最新

#### 3.2.8 移植性 (Portability)

| 副特性 | チェック内容 | 判定基準 |
|---|---|---|
| 適応性 | 環境変数で設定可能 | F: ハードコード, P: 環境変数化 |
| 設置性 | ワンコマンドデプロイ | F: 手動手順, W: 部分自動, P: 完全自動 |
| 置換性 | 外部サービスの差替可能性 | F: 密結合, W: 部分的, P: 抽象化済み |

検証ポイント:
- env.ts による Zod バリデーション（13環境変数）
- Docker / Vercel 両方でのデプロイ可能性
- 決済プロバイダ (Stripe) の差替容易性
- メール送信 (Resend) の差替容易性

## 4. Phase 3: FMEA リスク分析

Phase 2 で **Warn / Fail** となったセルに対して FMEA を適用し、リスクを定量化する。

### 4.1 RPN (Risk Priority Number) 算出

**RPN = 重大度(S) × 発生頻度(O) × 検出困難度(D)**

| スコア | 重大度 (S) | 発生頻度 (O) | 検出困難度 (D) |
|---|---|---|---|
| 1 | 影響なし | ほぼ発生しない | 即座に検出可能 |
| 2-3 | 軽微な不便 | 低頻度 | テストで検出可能 |
| 4-6 | 機能障害 | 中頻度 | 注意深い検査で検出 |
| 7-8 | データ損失 / 金銭影響 | 高頻度 | 特殊条件でのみ検出 |
| 9-10 | セキュリティ侵害 / 法的問題 | 常時 | 検出困難 |

### 4.2 RPN 閾値と対応方針

| RPN 範囲 | リスクレベル | 対応方針 |
|---|---|---|
| **≥200** | Critical | 即時対応必須（リリースブロッカー） |
| **100-199** | High | リリース前に対策完了 |
| **50-99** | Medium | バックログに積み、計画的に対応 |
| **<50** | Low | 受容可能（監視のみ） |

### 4.3 FMEA テーブルテンプレート

| ID | 故障モード | 影響 | S | 原因 | O | 現在の検出手段 | D | RPN | 推奨対策 | 対策状態 |
|---|---|---|---|---|---|---|---|---|---|---|
| FM-001 | (故障の内容) | (ユーザー/システムへの影響) | 1-10 | (根本原因) | 1-10 | (既存の検出/防止手段) | 1-10 | S×O×D | (具体的対策) | 未着手/設計済/実装済 |

## 5. Phase 4: ATAM シナリオ分析

アーキテクチャ上の重要トレードオフをシナリオベースで検証する。

### 5.1 ユーティリティツリー

品質属性を階層化し、各シナリオに **(重要度, 困難度)** を付与する。

```
システム品質
├── 性能
│   ├── (H,H) 調合リクエストが2秒以内に完了する
│   ├── (H,M) レシピ一覧が500ms以内に表示される
│   └── (M,L) 100同時ユーザーでp95<3秒を維持する
├── セキュリティ
│   ├── (H,H) 決済情報が第三者に漏洩しない
│   ├── (H,M) 他ユーザーのレシピを改ざんできない
│   └── (M,M) 管理者操作が全て追跡可能である
├── 可用性
│   ├── (H,M) Stripe障害時にカート機能は維持される
│   ├── (M,M) Redis障害時にAPI応答は継続する
│   └── (M,L) AI Service障害時に既存レシピ注文は可能
├── 変更容易性
│   ├── (H,M) 新しい香料を追加するのに1日以内で対応できる
│   ├── (M,M) 決済プロバイダを変更するのに1週間以内で対応できる
│   └── (M,L) 新しい調合モードを追加するのに3日以内で対応できる
└── スケーラビリティ
    ├── (H,L) ユーザー数10倍でもアーキテクチャ変更不要
    └── (M,L) クリエーター1000人でもティア計算が1時間以内に完了する
```

**(重要度, 困難度)**: H=High, M=Medium, L=Low
優先順位: (H,H) > (H,M) > (M,H) > (H,L) > (M,M) > ...

### 5.2 シナリオ詳細テンプレート

```
シナリオID: AS-XXX
品質属性: (性能 / セキュリティ / 可用性 / 変更容易性 / スケーラビリティ)
シナリオ: (刺激 → 環境 → 応答 → 測定指標)
現在の設計判断: (採用したアプローチ)
代替案: (検討した他のアプローチ)
トレードオフ: (何を犠牲にして何を得たか)
感度点: (パラメータ変更で結果が大きく変わる箇所)
リスク: (未解決のリスク)
```

### 5.3 トレードオフ分析

| ID | シナリオ | 関連品質属性 | 現在の判断 | 代替案 | トレードオフ | リスク |
|---|---|---|---|---|---|---|
| AS-001 | 調合をリアルタイム計算 vs キュー | 性能 vs UX | リアルタイム | ジョブキュー + ポーリング | 即時応答だが高負荷時タイムアウト | AI Service のスケーリング必要 |
| AS-002 | セッション: JWT vs Server Session | 性能 vs セキュリティ | NextAuth (JWT) | Redis Session Store | ステートレスだがトークン即時無効化不可 | リフレッシュトークン + 短い有効期限で緩和 |
| AS-003 | レシピ配合比率: 正規化 vs 非正規化 | 保守性 vs 性能 | 正規化 (recipeFlavors) | JSON カラムに全配合埋め込み | 柔軟だが JOIN 必要 | キャッシュで緩和 |
| AS-004 | 通知: ポーリング vs SSE/WebSocket | コスト vs UX | ポーリング (30s) | Server-Sent Events | 安価だが遅延最大30秒 | Phase 2 で SSE 移行可能な設計 |
| AS-005 | 検索: pg_bigm vs Elasticsearch | 運用コスト vs 検索品質 | PostgreSQL pg_bigm | Elasticsearch クラスタ | 運用シンプルだが大規模時性能劣化 | レシピ数1万件超で再評価 |
| AS-006 | 画像保存: Vercel Blob vs S3 | 運用容易性 vs 柔軟性 | Vercel Blob | S3 + CloudFront | ロックインだが設定不要 | 容量増大時のコスト |

### 5.4 感度点 (Sensitivity Points)

アーキテクチャの判断が品質に大きく影響するパラメータ。

| 感度点 | 関連シナリオ | 影響 | 監視方法 |
|---|---|---|---|
| AI Service 応答時間 | AS-001 | 調合の UX に直結 | p95 レイテンシ監視 |
| DB 接続プール max サイズ | FM-003 | 同時接続数の上限 | プール使用率アラート |
| Redis キャッシュ TTL | AS-003 | 鮮度 vs 負荷のバランス | キャッシュヒット率 |
| レート制限閾値 | FM-002 | セキュリティ vs ユーザビリティ | 429 エラー率 |
| Stripe Webhook タイムアウト | FM-001 | 決済処理の信頼性 | Webhook 失敗率 |

### 5.5 トレードオフグラフ（定性評価）

```
         性能 ◄──────── 現在の設計 ────────► セキュリティ
               ├─ リアルタイム調合(↑性能)
               ├─ JWT セッション(↑性能)
               ├─ レート制限(↑セキュリティ, ↓性能)
               └─ 監査ログ(↑セキュリティ, ↓性能)

      変更容易性 ◄──────── 現在の設計 ────────► 性能
               ├─ 正規化 DB(↑変更容易性, ↓性能)
               ├─ サービス層分離(↑変更容易性)
               └─ PostgreSQL 検索(↑運用容易性, ↓検索性能)

      コスト ◄──────── 現在の設計 ────────► UX
               ├─ ポーリング通知(↓コスト, ↓UX)
               ├─ Vercel Blob(↓コスト, ↓柔軟性)
               └─ pg_bigm(↓コスト, ↓検索品質)
```

### 6.1 実行計画

| Phase | 手法 | 並列度 | 入力 | 出力 |
|---|---|---|---|---|
| 1 | C4 構造検証 | 4並列 (L1-L4) | ソースコード + 設計書 | 乖離リスト |

| 3 | FMEA | 1 (集約) | Phase 2 の W/F セル | RPN ランク付き故障モード一覧 |
| 4 | ATAM | 1 (集約) | Phase 1-3 結果 | トレードオフ分析レポート |

- Phase 1, 4 は独立実行可能
- Phase 2 は Phase 1 完了後
- Phase 3 は Phase 2 完了後

### 6.2 エージェント並列実行構成

```
Phase 1 (並列):
  Agent-C4-L1: Context レベル検証
  Agent-C4-L2: Container レベル検証
  Agent-C4-L3: Component レベル検証
  Agent-C4-L4: Code レベル検証

Phase 2 (並列):
  Agent-ISO-A: 機能適合性 + 性能効率性
  Agent-ISO-B: 互換性 + 使用性
  Agent-ISO-C: 信頼性 + セキュリティ

  Agent-ATAM: シナリオ分析 + トレードオフ検証
```
